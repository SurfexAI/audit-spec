# Channel 6: A2A Agent Card

## What this channel measures

Whether your product publishes an Agent-to-Agent (A2A) protocol agent card describing its capabilities, endpoints, and interaction patterns in a format other agents can consume directly. A2A is the protocol for agent-to-agent communication, distinct from MCP which is primarily agent-to-tool. An A2A agent card declares "here is what I can do as an agent, and here is how to invoke me."

## Why it matters

A2A is the emerging standard for products that operate as agents themselves rather than as passive tools. The protocol is backed by Google, Salesforce, SAP, and a coalition of 50+ partners. As multi-agent workflows become common, products that present themselves as agents through an A2A card become composable participants in those workflows. Products without an agent card are limited to being invoked as tools, not as collaborators. This channel is forward-looking: A2A adoption is earlier than MCP, but the protocol is converging on standard locations and formats fast enough that absence will become conspicuous within the next year.

## The four tiers

**Not Present**
No A2A agent card exists. The product cannot be discovered or invoked as an agent participant in A2A workflows.

**Present but Deficient**
An agent card exists but is incomplete or improperly formatted. Common patterns: card is published at a non-standard URL with no DNS pointer; required fields (name, description, capabilities) are present but skills/actions are vague; card references endpoints that return errors; authentication mechanism is documented but not implemented; card has not been updated since the product's capabilities changed.

**Functional**
An A2A agent card is published at a discoverable location, contains the required fields per the A2A specification, and accurately describes the product's agent-callable capabilities. Skills and actions are documented with input and output schemas. Authentication is documented and works. Another agent can read the card and successfully invoke at least one skill.

**Optimized**
The agent card is published at the standard location (/.well-known/agent.json or equivalent per current A2A spec), includes complete skill definitions with input schemas, output schemas, examples, and error responses. The card declares supported transport mechanisms, authentication options, and capability versioning. The product participates in A2A discovery registries where they exist. Skills are scoped to clear agent-meaningful tasks rather than thin wrappers around REST endpoints.

## Observable signals

The crawler checks for:

- Agent card at /.well-known/agent.json on the product's primary domain
- Agent card at /agent.json or /a2a.json as fallback locations
- Valid JSON response matching the A2A agent card schema
- Required fields populated: name, description, version, capabilities or skills
- Skill definitions include input and output schemas
- Authentication mechanism documented in the card
- Endpoints referenced in the card return non-error responses to spec-conformant probes
- DNS TXT record or HTTP header advertising A2A support
- Listing in A2A directory or registry where one exists

## Common failure modes

- Product team is aware of A2A but waiting for the spec to stabilize before publishing
- Agent card exists but is generated from internal documentation and lists every API endpoint as a separate skill, producing a card too dense for agents to use
- Skills are defined but lack input or output schemas, leaving agents unable to construct valid invocations
- Card is published at a documented URL but not at the well-known location, requiring agents to know where to look
- Authentication mechanism documented in the card differs from what the actual endpoints require
- Card was generated once and never updated as the product evolved
- Skills duplicate MCP tool definitions verbatim without adapting for the A2A interaction model

## What an agent does with this channel

An agent operating in a multi-agent workflow queries discoverable agent cards to find collaborators that can handle subtasks. If your card declares a skill matching the subtask, the agent invokes your endpoints using the documented schema and authentication. The agent treats your product as a peer in the workflow, sending structured requests and consuming structured responses. A missing or broken agent card means your product is not considered for multi-agent collaboration, regardless of how capable your REST API is.

## Note on protocol maturity

A2A is younger than MCP and the spec continues to evolve. The current audit checks against the published A2A specification as of the audit date. Products that publish a card matching an older draft of the spec may score lower than their actual readiness warrants. As the spec stabilizes, this channel's signals will tighten.
