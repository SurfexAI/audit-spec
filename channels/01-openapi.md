# Channel 1: OpenAPI / Swagger Specification

## What this channel measures

This channel measures whether an autonomous AI agent can read a complete, accurate, machine-readable specification of your API. The OpenAPI specification (formerly known as Swagger) is the de facto standard for describing REST APIs in a format that both humans and machines can parse.

When an agent attempts to integrate with your product, the first question it asks is: what endpoints exist, what parameters do they accept, what do they return, and how is authentication handled? An OpenAPI spec answers all of these questions in a single file. Without one, an agent has to scrape your documentation, infer endpoint structures from examples, and guess at edge cases. Most agents will fail or take far longer to integrate.

## Why it matters

Agents do not read your developer documentation the way a human developer does. They look for structured, parseable specifications first and fall back to unstructured docs only when necessary. A complete OpenAPI spec at a discoverable URL is the single highest-leverage signal that your API was built with agents in mind.

The specification also enables an entire class of downstream tooling: SDK generation, automated testing, mock servers, API gateways, and MCP server scaffolding. A company without an OpenAPI spec is invisible to most of this ecosystem.

## The four tiers

### Not Present

No OpenAPI or Swagger specification exists anywhere on the company's public surfaces. Agents must rely on scraping unstructured documentation if any documentation exists at all.

### Present but Deficient

A specification exists but is hard to find, incomplete, or out of date. Common patterns:

- Spec lives only inside a GitHub repository's docs/ folder rather than at a discoverable public URL
- Spec is hosted on a third-party documentation CDN (Speakeasy, Mintlify, ReadMe) without a stable canonical URL
- Spec covers only a small subset of the actual API surface
- Spec exists for an older API version that no longer matches the live endpoints
- Spec is referenced in documentation but the linked file returns 404 or 500

### Functional

A reasonably complete OpenAPI 3.x specification is available at a stable public URL, covers the major endpoints, and matches the live API. The spec may have minor gaps but an agent can complete a typical integration using it as the primary reference.

Common signals of a Functional spec:

- Hosted at a predictable URL such as /openapi.yaml, /openapi.json, /api/openapi.json, or /.well-known/openapi.json
- Linked from the company's developer documentation
- Major endpoints covered with request/response schemas
- Authentication scheme defined

### Optimized

A complete, current, machine-readable OpenAPI 3.x specification is available at a stable public URL and reflects the live API exactly. Agents can integrate using the spec alone, without referencing additional documentation.

Common signals of an Optimized spec:

- Hosted at a standardized URL (/openapi.yaml, /openapi.json, or /.well-known/openapi.json)
- Referenced in llms.txt or the company's machine-readable surface map
- Updated automatically when the API changes (versioned alongside the codebase)
- All endpoints, parameters, response codes, and error formats documented
- Authentication, rate limits, and pagination fully specified
- Includes example requests and responses
- Validates against the OpenAPI 3.x schema with no errors

## Observable signals

The Surfex crawler first resolves the company's canonical domain: it fetches the homepage and follows redirects, and when the landing host differs from the input host it probes standard paths on host candidates derived from both domains, with input-domain results keeping priority. It then looks for the OpenAPI specification at these locations in order:

1. /.well-known/openapi.json and /.well-known/openapi.yaml
2. /openapi.json and /openapi.yaml
3. /api/openapi.json and /api/openapi.yaml
4. URLs referenced in llms.txt
5. URLs linked from developer documentation pages on the input domain, the canonical domain, and docs-family subdomains such as docs.{domain}
6. Specifications hosted on common documentation CDNs (Speakeasy, Mintlify, ReadMe, Bump)

When a candidate spec is found, the crawler validates that it parses as OpenAPI 3.x, counts the documented endpoints, and checks whether the spec covers the endpoints referenced in the company's public documentation.

## Common failure modes

- Spec exists but is hidden inside a GitHub repository's docs/ folder rather than served from a public URL. This is the single most common pattern observed across audited dev tool companies. The fix is to host the spec at a stable public URL on the documentation site.
- Spec is generated by a documentation platform and lives at an unpredictable URL such as a hashed CDN path that changes between deploys. Even if the spec is technically public, agents cannot discover it without a stable canonical location.
- Spec covers only some endpoints. Often the case when documentation is hand-written for some endpoints and the spec is generated for others. Agents that find the spec assume it is complete and miss the undocumented endpoints.
- Spec drifts from the live API. Common when the spec is hand-maintained rather than generated from the codebase. Agents integrate against documented behavior and fail at runtime when the live API differs.
- Spec exists at /openapi.json but the API itself lives at a different subdomain without redirection or servers configuration in the spec. Agents read the spec but call the wrong base URL.

## What an agent does with this channel

When an agent attempts to integrate with a product, it typically:

1. Looks for /.well-known/openapi.json first
2. Falls back to common standardized paths (/openapi.json, /openapi.yaml)
3. Reads llms.txt to see if a specification URL is referenced there
4. Parses the company's developer docs to find a spec link
5. Falls back to scraping unstructured docs only if no spec is found

A spec at a discoverable URL turns a multi-step integration attempt into a single API call to the agent's parsing toolchain. Without one, integration takes longer, fails more often, and produces less reliable code.
